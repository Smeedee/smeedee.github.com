Getting started

Our Getting Started guide is written with the aim to help new developers to quickly get started contributing through Smeedee development.

1. Set up your development environment

See Development tools for a list of what you need.

2. Check out the source code

We use SubVersion for Source Control Management. For Windows users a good client is TortoiseSVN, however, any client of your choice can be used.

You can check out or Subversion repository from http://smeedee.googlecode.com/svn/trunk

3. Open the main solution

In the checked out folder, you can find our project in the trunk/source folder. In this folder you will find the solution file Smeedee.sln which contains all the projects needed for Smeedee. Smeedee.Client.Web is the main entrance to the smeedee application and should be used as start up project.

4. What you can expect to find in the solution

Smeedee's Visual Studio solution is segmented into the following parts:

  • Buildscripts contains the scripts necessary to build Smeedee, run tests and generate setup kits.
  • Client contains client specific parts such as the client application itself and the different widgets, all enforcing the MVVM pattern.
  • Common contains common framework across the application such as settings and configuration.
  • DomainModel contains the domain concepts that Smeedee's different tasks and widgets are based on.
  • Scheduler/Tasks contains the different tasks that are run server side such as those used for gathering data from third party systems as well as the system for scheduling and dispatch these tasks.

Most parts consists of three projects:

  • 1 project for production code
  • 1 silverlight project for the view
  • 1 project with tests

5. Testing

We are doing Test Driven Development (TDD) and strongly encourage you to do the same as this will help make sure the source code easier to maintain as well as minimizing technical debt and acting as documentation.

We are also using TinyBDD - A behaviour driven development (BDD) DSL for C# which is also test runner agnostic and lets you do BDD syntax while writing tests.
